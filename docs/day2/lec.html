Linear systems in R; solution spaces

Day 2 — Quantitative biology

Quantitative biology and integrative omics

About this lecture

Yesterday we solved small systems \(A\mathbf{x} = \mathbf{b}\) by hand. Today we hand the work to R — and discover that not every system has a tidy answer. You will be able to:

  • Build \(A\) and \(\mathbf{b}\) in R and solve with solve.
  • Recognise the three kinds of solution space: exactly one, none, or infinitely many.
  • Connect each kind to a deconvolution scenario: exactly-, over-, or under-determined.
  • Know what to reach for when \(A\) is singular or non-square: least squares, via lm (full treatment later).

Today is the R day. The key conceptual jump is that “solve a linear system” is not always a well-posed request — there may be no answer, or a whole family of answers. We tie that to whether a cell-type deconvolution problem has enough, too few, or conflicting measurements. Keep the code minimal and the intuition front and centre.

Recap: the deconvolution problem

A bulk expression sample is a mixture. Known per-cell-type signatures (matrix \(A\), one column per cell type) and the measured bulk profile (vector \(\mathbf{b}\)) give a system \[ A\mathbf{x} = \mathbf{b}, \] whose solution \(\mathbf{x}\) is the fractions (or amounts) of each cell type. Today: what if the system cannot be solved exactly?

Recall the running example. The number of measured genes is the number of equations (rows of A); the number of cell types is the number of unknowns (columns of A). The relationship between those two counts is what drives everything today.

Building \(A\) and \(\mathbf{b}\) in R

# Two cell types, two marker genes
A <- matrix(c(2, 1,
              1, 3), nrow = 2, byrow = TRUE)
b <- c(8, 9)

A
#      [,1] [,2]
# [1,]    2    1
# [2,]    1    3
  • matrix with byrow = TRUE fills row by row, so the code reads like the equations on paper.
  • nrow sets the number of rows; c(...) is just the entries.
  • Default byrow = FALSE fills column by column — a classic bug.

Spend a moment on byrow because it bites everyone once. The default fills down the first column, so a matrix typed left-to-right comes out transposed unless you set byrow = TRUE. Always print the matrix and eyeball it against your equations before trusting it.

Solving with solve()

x <- solve(A, b)
x
# [1] 3 2          # 3 units tumour, 2 units immune

# Always check: does A x reproduce b?
A %*% x
#      [,1]
# [1,]    8
# [2,]    9
  • solve(A, b) returns the \(\mathbf{x}\) with \(A\mathbf{x} = \mathbf{b}\) — it runs Gaussian elimination, exactly what you did by hand on day 1.
  • %*% is matrix multiplication. Use it to verify, just like substituting back on paper.

solve is the workhorse. Two arguments means “solve this system”. Note that it returns the same 3, 2 we found by hand yesterday. The verification step with the matrix product is the R equivalent of plugging back into the original equations — make it a reflex.

solve(A) alone gives the inverse

Ainv <- solve(A)        # one argument: the inverse of A
Ainv %*% b
# [1] 3 2               # same answer, x = A^{-1} b
  • solve(A, b) is preferred over solve(A) %*% b: it is faster and more numerically accurate (it never forms the inverse explicitly).
  • The inverse \(A^{-1}\) only exists when \(A\) is square and non-singular — precisely the “exactly one solution” case below.

Worth showing so they recognise both forms in the wild, but steer them toward the two-argument solve. Forming the inverse is wasteful and less stable. Flag that the inverse existing is equivalent to the system having a unique solution — this links directly into the next slides.

Three kinds of solution space

For \(A\mathbf{x} = \mathbf{b}\) there are exactly three possibilities:

  1. Exactly one solution — the system is consistent and determined.
  2. No solution — the equations contradict each other.
  3. Infinitely many solutions — the equations are consistent but leave freedom.

The next slides take each in turn and pair it with a deconvolution scenario.

This trichotomy is a theorem: a linear system has zero, one, or infinitely many solutions — never, say, exactly two. Geometrically (day 3) each equation is a line or plane, and we are asking how they intersect. For now, anchor each case to the biology.

Case 1 — exactly one (well-determined)

As many independent measurements as unknowns. \(A\) is square and non-singular.

Biology: #genes measured = #cell types, and the signatures are genuinely different. The deconvolution has a single answer.

A <- matrix(c(2, 1,
              1, 3), nrow = 2, byrow = TRUE)
solve(A, c(8, 9))
# [1] 3 2          # the unique solution

This is the happy case from day 1. solve() just returns the answer.

This is the case students already understand from yesterday. Stress the two conditions: square (equations equal unknowns) and non-singular (the equations are genuinely independent, not rephrasings of each other). When both hold, there is one and only one solution and solve returns it.

Case 2 — no solution (over-determined)

More equations than unknowns, and they conflict. Often \(A\) is non-square (more rows than columns).

Biology: we measured more genes than cell types. With real noisy data the equations rarely agree exactly, so no \(\mathbf{x}\) fits all of them.

A <- matrix(c(1, 0,
              0, 1,
              1, 1), nrow = 3, byrow = TRUE)   # 3 eqns, 2 unknowns
b <- c(2, 3, 6)                                # but 2 + 3 = 5, not 6
solve(A, b)
# Error in solve.default(A, b) : 'a' must be a square matrix

solve() refuses: it only handles square systems.

Over-determined means too many equations. With exact data they might still all agree, but with measurement noise they generally do not — there is no x that satisfies every equation at once. solve flatly errors on a non-square A. The fix, least squares, is two slides away. Note this is actually the realistic omics case: thousands of genes, a handful of cell types.

Case 3 — infinitely many (under-determined)

Fewer independent equations than unknowns. \(A\) is singular (or has fewer rows than columns).

Biology: we measured too few genes to distinguish the cell types — many fraction combinations explain the data equally well.

A <- matrix(c(1, 1,
              2, 2), nrow = 2, byrow = TRUE)   # row 2 = 2 x row 1
b <- c(3, 6)
solve(A, b)
# Error in solve.default(...) : Lapack routine dgesv: system is exactly singular

Both equations say the same thing (\(x_1 + x_2 = 3\)), so any \(\mathbf{x}\) on that line works. solve() errors on the singular matrix.

Under-determined means too few independent equations. Here row two is just twice row one — no new information — so the matrix is singular and there is a whole line of solutions. solve detects the singularity and stops. The biology: not enough markers to resolve the cell types. Contrast the two error messages on this and the previous slide so they can diagnose which case they hit.

Detecting which case you are in

dim(A)                 # rows (equations) vs cols (unknowns)
det(A)                 # square only: ~0 means singular
qr(A)$rank             # number of independent equations
  • dim non-square \(\Rightarrow\) over- or under-determined by shape.
  • det \(\approx 0\) (square) \(\Rightarrow\) singular \(\Rightarrow\) not Case 1.
  • qr$rank < #unknowns \(\Rightarrow\) the equations do not pin down every unknown.

A square \(A\) with \(\det A \ne 0\) and full rank is exactly Case 1.

Give them a diagnostic toolkit. dim tells you the shape. det near zero flags a singular square matrix — but careful, floating point means it is rarely exactly zero, so “small relative to the entries” is the real test. Rank is the most robust single number: full rank equal to the number of unknowns is the unique- solution condition.

When there is no exact answer: least squares

For over-determined systems (Case 2) we cannot satisfy every equation, so we find the \(\mathbf{x}\) that comes closest — minimising the total squared mismatch \(\lVert A\mathbf{x} - \mathbf{b}\rVert^2\). This is least squares.

A <- matrix(c(1, 0,
              0, 1,
              1, 1), nrow = 3, byrow = TRUE)
b <- c(2, 3, 6)

qr.solve(A, b)            # least-squares solution
# [1] 2.333333 3.333333

coef(lm(b ~ A - 1))       # same thing, via linear regression

qr.solve and lm both give the best-fit \(\mathbf{x}\). We treat this properly later in the course.

This is the bridge to the rest of the course. When you cannot fit everything, fit as well as possible by minimising squared error — that is least squares, and it is what lm does under the hood. Mention it lightly here; the geometry (projection, day 3) and the full regression treatment come later. The takeaway: an over-determined deconvolution is solved by least squares, not solve.

A note for real deconvolution

Real cell-type fractions must be non-negative and usually sum to 1. Plain solve/lm know nothing about that and can return negative fractions.

  • Production tools (e.g. CIBERSORT, and Bioconductor packages) add these constraints (non-negative least squares, support-vector regression, etc.).
  • The linear-system core is the same — the constraints are extra structure on top.

We use the unconstrained version to learn the mechanics; the constrained version is the real tool.

Honesty about what we defer, per the course design principles. The bare linear algebra can hand you a negative fraction, which is biologically nonsense. Real deconvolution adds constraints. We deliberately strip those away to expose the linear-system skeleton students need to learn first. Name-drop the real tools so they know where to look.

Summary

  • Build systems in R with matrix (byrow = TRUE!) and solve with solve(A, b); check with %*%.
  • A linear system has one, none, or infinitely many solutions — matching well-, over-, and under-determined deconvolution.
  • Diagnose with dim, det, qr$rank.
  • Over-determined \(\Rightarrow\) least squares (qr.solve, lm); full treatment later.
